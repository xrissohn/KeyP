KeyP는 “모든 키워드를 모든 곳에서 계속 긁는 앱”이 아니라, AlphaGo처럼 먼저 가능성이 높은 후보 경로를 좁힌 뒤에만 비용을 쓰는 앱이어야 한다. 제품적으로는 관심사 해석 → 소스 우선순위화 → 저비용 수집 → 검증 → 요약/전달 → 피드백 학습 → 내부 상호매칭 순서가 가장 리소스 효율적이며, 이 구조는 AlphaGo의 정책 기반 후보 축소, contextual bandit의 탐색-활용 최적화, reciprocal recommender의 양방향 매칭 논리를 제품 설계로 번역한 형태에 가깝다. 

방향
KeyP의 핵심 단위는 키워드가 아니라 관심사 객체여야 한다. 즉 사용자의 자연어 입력을 그대로 검색 문자열로 던지는 것이 아니라, 무엇을, 왜, 어디서, 언제까지, 어떤 형태의 신호를, 어느 정도 신뢰도로, 누구와 연결되길 원하는지를 구조화한 InterestSpec으로 먼저 바꿔야 한다. 그래야 “다음 주 여행”, “앞으로 뜰 사람”, “자주 정확하게 알려주는 인플루언서”, “내 관심사와 서로 맞는 사람”처럼 미래지향적이고 문맥 의존적인 요청을 다룰 수 있고, 이후 소스 선택도 키워드가 아니라 맥락 기반 확률 문제로 바뀐다. 이런 접근은 클릭/반응 피드백을 보며 다음 액션을 고르는 contextual bandit과, 사람-대-사람 후보를 단순 검색이 아니라 상호 적합성 문제로 보는 reciprocal recommendation의 방향과 맞닿아 있다. 

이 때문에 KeyP의 차별점은 “검색 결과를 많이 보여주는 것”이 아니라 미래 가능성이 높은 정보 흐름을 먼저 감지하고, 동시에 KeyP 내부의 사용자 관심사를 서로 연결해주는 것이어야 한다. 외부 소스 탐색과 내부 상호매칭을 분리하되, 사용자-사용자 연결은 외부 웹 탐색보다 먼저 계산하는 편이 낫다. 내부 그래프는 API 비용이 거의 들지 않고, 상호관심·상호조건·상호동의까지 계산할 수 있어 결과 품질이 높아지기 때문이다. 이 부분은 사람을 다른 사람에게 추천하는 reciprocal recommender의 핵심 전제와 정확히 일치한다. 

제품 범위
KeyP는 “실시간 정보 검색 앱”이 아니라 실시간 사용자 관심사 알림 및 상호매칭 앱으로 정의하는 편이 맞다. 외부 세계에서는 새로운 게시물, 댓글, 라이브, 업로드, 뉴스, 토론, 트렌드, 행사 예고, 커뮤니티 반응 같은 신호를 모으고, 내부 세계에서는 사용자들이 등록한 관심사를 서로 구조화해 잠재적 연결을 만든다. 이 이중 구조가 있어야 사용자가 적을 때는 외부 소스 중심으로 가치가 나오고, 사용자가 많아질수록 내부 매칭과 집단 피드백이 가치의 중심이 되는 네트워크 효과가 생긴다. 

다만 사람을 찾는 기능은 외부 플랫폼의 개인 정보를 캐내는 방식이 아니라 KeyP 내부의 명시적 opt-in 상호매칭으로 설계해야 한다. Apple의 심사 가이드라인은 UGC·소셜 앱에 필터링, 신고, 차단, 운영자 연락수단을 요구하고, 주로 포르노성·랜덤/익명 채팅·실존 인물 대상 objectification으로 흘러가는 앱은 스토어에 적합하지 않다고 밝힌다. Google Play도 UGC 앱에 대해 지속적 모더레이션, 신고, 차단, 약관 동의, 미성년자 안전장치를 요구한다. 따라서 “외부 데이팅앱을 뒤져 특정 집단의 사람을 찾아주는” 방향보다, KeyP 내부에서 서로의 의도와 조건을 명시적으로 등록하고 양쪽이 동의했을 때만 연결되는 구조가 훨씬 안전하고 배포 가능성이 높다. 

실시간 소스 전략
외부 공개 신호를 넓고 빠르게 잡는 1차 축은 YouTube, X, Reddit가 가장 실용적이고, Instagram과 TikTok는 연결 방식이 더 제한적이므로 “전체 인터넷 폭넓은 탐색”의 1순위보다는 인플루언서·크리에이터 opt-in/연결형 보강 소스로 두는 편이 맞다. Instagram Platform은 professional account 중심으로 열려 있고, TikTok Display API 역시 크리에이터의 프로필·영상 표시용이며 웹훅과 권한 승인이 중심이다. TikTok Research Tools는 자격 있는 연구자 대상이라 일반 상용 앱의 기본 수집선으로 보기 어렵다. 

YouTube는 검색 자체보다 채널 기반 감시가 훨씬 효율적이다. search.list는 호출당 100 quota units를 쓰고, 기본 일일 할당량은 10,000 units이므로 “키워드별 실시간 무한 검색” 구조는 금방 비싸진다. 반대로 YouTube Data API는 PubSubHubbub 기반 푸시 알림을 지원해 채널의 업로드·제목 수정·설명 수정 이벤트를 웹훅처럼 받을 수 있다. 따라서 KeyP는 먼저 관심 주제와 관련된 채널·플레이리스트·작성자 집합을 학습하고, hot path는 채널 구독/푸시로 처리하고, cold start나 신규 관심사에서만 제한적으로 검색 API를 써야 한다. 

X는 빠른 반응성과 대화 흐름 추적에 강하지만 비용과 범위 제한을 명확히 의식해야 한다. 최근 검색 엔드포인트는 지난 7일 데이터만 다루고, 전체 firehose나 고볼륨 스트림은 enterprise 영역이다. 현재 X API는 pay-per-usage 구조이고, 엔드포인트별 rate limit이 있으며, 개발자 가이드라인은 스크래핑이나 비공식 수단 금지, 공식 API 사용, rate limit 준수, off-X matching에는 opt-in consent 필요를 명시한다. 따라서 KeyP에서 X는 “고빈도 hot source”가 될 수는 있어도, 질문 분석 후 가장 가능성 높은 토픽·작성자·연산자만 고르는 비용 민감형 라우팅 소스로 다뤄야 한다. 

Reddit은 서브레딧 구조 때문에 니치 관심사 탐지에 매우 좋고, 공식 문서도 listing 데이터가 자주 변한다고 설명한다. 특히 live threads는 반복 polling보다 websocket 수신이 더 낫다고 직접 안내한다. 따라서 KeyP에서는 new·rising·search 중심의 커뮤니티 감시와, 몇몇 초핵심 라이브 스레드의 websocket 구독을 분리해 쓰는 전략이 적절하다. 이런 구조라면 실시간성은 살리면서도 전체 플랫폼을 무식하게 재조회하는 비용을 줄일 수 있다. 

멀티 에이전트 구조
KeyP의 하네스는 다섯 단계보다 일곱 단계가 더 낫다. 총괄 사고 에이전트는 자연어를 InterestSpec으로 구조화하고, 요청의 의도 유형을 monitor, alert, opportunity, match, creator_watch, travel, local_signal처럼 라벨링한다. 수요 클러스터링 에이전트는 서로 비슷한 관심사를 한 묶음으로 만들어 동일한 외부 수집 작업을 여러 사용자에게 재사용하게 한다. 소스 라우터 에이전트는 어떤 플랫폼·어떤 작성자·어떤 연산자가 지금 가장 성공 확률이 높은지 prior를 계산한다. 스카웃 에이전트는 실제 수집을 수행하고, 검증 에이전트는 중복 제거·출처 교차검증·신뢰도 산정을 수행하며, 상호매칭 에이전트는 KeyP 내부 사용자들 사이의 양방향 적합성을 계산한다. 마지막으로 전달 에이전트는 요약·링크·푸시를 만들고, 학습 에이전트는 피드백으로 가중치를 갱신한다. 이런 분업은 확률 기반 후보 축소와 온라인 피드백 학습에 잘 맞고, Replit의 Agents & Automations 문서가 말하는 event-driven/scheduled automation 패턴과도 호응한다. 

이 구조의 핵심은 비싼 작업을 뒤로 미루는 것이다. KeyP는 검색을 시작하기 전에 먼저 “이 관심사라면 어느 소스군이 가장 잘 맞는가”를 추정해야 한다. 추천 점수는 실무적으로 예상보상 + 탐색보너스 + 소스신뢰도 + 작성자친화도 + 상호매칭가중치 - API비용 - 중복패널티 같은 형태가 적절하다. 즉 AlphaGo식으로 후보를 줄이고, contextual bandit식으로 불확실하지만 잠재력이 높은 소스를 일부 탐색하며, reciprocal system처럼 양쪽의 조건이 맞을 때만 사람 추천 점수를 올리는 식이다. 이런 방식이어야 “리소스 최소화 + 속도 최대화 + 품질 학습”이 동시에 가능하다. 

수요 클러스터링 에이전트는 특히 중요하다. 예를 들어 비슷한 시기·비슷한 장소·비슷한 주제의 관심사가 동시에 많이 들어오면, 외부 소스 수집은 하나의 shared job으로 처리하고 결과 fan-out만 사용자별로 다르게 하는 편이 토큰과 API 비용을 크게 줄인다. 사용자가 늘수록 수집 공통분모가 커지고, 공통 수집 뒤 개인화 재정렬만 하면 되므로 시스템은 오히려 더 효율적으로 학습한다. 사용자가 많아질수록 검색 빈도와 품질이 같이 올라간다는 당신의 가정은 바로 이 shared retrieval + 개인화 rerank 구조에서 가장 현실적으로 구현된다. 

랭킹과 학습
랭킹은 정적 규칙보다 contextual bandit + online reranking 조합이 맞다. 첫 단계에서는 관심사 유형, 장소, 시간창, 최근 반응, 과거 정답률, 사용자가 선호한 매체/작성자 이력을 특징으로 넣어 각 소스군의 기대보상을 예측한다. 둘째 단계에서는 후보 알림 카드들 사이에서 relevance·freshness·trust·author affinity·reciprocity를 재정렬한다. KeyP가 받게 되는 피드백은 대부분 전체 문서 정답 라벨이 아니라 “열어봤다 / 저장했다 / 링크를 눌렀다 / 좋아요 / 별로예요 / 이 소스는 숨기기 / 같은 스타일 더 보기” 같은 제한적 top-k 피드백이므로, 온라인 랭킹은 이런 부분관측 학습에 맞춰 설계해야 한다. 

매체·인플루언서 우선선별은 “팔로워 수”보다 문맥별 정밀도를 기준으로 두는 편이 낫다. 예를 들어 사용자가 뉴욕 여행 + 한식 + 현지 반응 + 다음 주 같은 관심을 등록했을 때, 과거에 비슷한 컨텍스트에서 좋은 피드백을 많이 받은 작성자·채널·커뮤니티는 prior가 올라가고, 클릭은 많지만 실제 만족도가 낮은 소스는 내려가야 한다. 즉 KeyP는 “인기순 추천”이 아니라 “이 맥락에서 자주 빠르고 정확했던 사람/매체”를 우선시해야 한다. 이때 explicit feedback과 implicit feedback을 함께 쓰는 것이 중요하며, 최근 온라인 ranking 연구도 다층 피드백을 활용하는 편이 더 낫다고 본다. 

검증은 두 단계로 두는 편이 안정적이다. 첫째, canonical event 중복 제거를 한다. 같은 영상, 같은 포스트, 같은 기사, 같은 커뮤니티 루머를 플랫폼 표현만 다른 별개 결과처럼 보내면 신뢰가 무너진다. 둘째, 신뢰도 검증을 한다. 하나의 고신뢰 소스가 직접적 증거를 제공하면 단독 통과시킬 수 있고, 저신뢰 소스라면 두 개 이상 독립 출처의 교차확인을 요구하는 식이 좋다. 알림 카드에는 왜 지금 이 알림이 떴는지, 얼마나 빠른 신호인지, 얼마나 검증되었는지, 원문 링크가 무엇인지를 같이 보여줘야 사용자가 피드백을 줄 수 있고 모델도 더 빨리 학습한다. 

Replit 구현안
2026년 현재 Replit은 Agent로 네이티브 모바일 앱을 만들고, iOS Simulator/Android Emulator 또는 실기기에서 미리보기하고, 전체 백엔드까지 함께 붙이는 흐름을 공식적으로 지원한다. 또 React Native + Expo 조합을 모바일 기본 흐름으로 안내하고, full-stack 모바일 앱에 Database, Object Storage, Connectors, AI integrations를 함께 붙일 수 있다고 명시하고 있다. 따라서 KeyP는 “웹 먼저”보다 React Native + Expo 기반 네이티브 모바일 우선으로 가는 편이 현재 Replit의 강점과 가장 잘 맞는다. 

구현 스택은 다음 구성이 가장 현실적이다. 모바일 클라이언트는 Expo React Native + Expo Router + expo-notifications를 쓰고, 서버는 TypeScript 기반 API 서버로 분리한다. 백엔드는 항상 살아 있어야 하는 수집 워커·웹소켓·오케스트레이터 때문에 Reserved VM이 적합하고, 주기적 재수집이나 배치 재랭킹은 Scheduled Deployments에 태우는 편이 좋다. 데이터는 Replit SQL Database에 저장하고, 원문 스냅샷·썸네일·첨부 파일은 App Storage에 두며, 비밀값은 Secrets로 분리한다. published app의 파일시스템에 영속 데이터를 의존하지 말라는 Replit 문서의 권고도 그대로 따르는 것이 안전하다. 

인증은 베타 내부 툴이라면 Replit Auth만으로도 시작할 수 있지만, 일반 소비자용 앱이라면 Clerk Auth 쪽이 더 맞다. Replit Auth는 사용자가 Replit 계정으로 로그인하는 구조이고, Clerk Auth는 독립 계정과 커스텀 브랜딩, 다양한 SSO 제공자를 위한 전용 인증 시스템을 제공한다. KeyP처럼 외부 사용자 대상 모바일 서비스라면 기본값은 Clerk Auth, 초기 데모/베타에는 Replit Auth 또는 private deployment를 병행하는 하이브리드가 가장 무난하다. 

푸시 알림은 Expo 문서 기준으로 iOS와 Android 알림 흐름을 상당 부분 단순화할 수 있고, 서버 측에서도 배송 실패·에러 처리를 포함한 신뢰성 설계를 권장한다. 또한 배터리와 전력 최적화를 위해 background task는 expo-background-task와 expo-task-manager 계열을 쓰는 편이 낫다. 즉 KeyP는 “앱이 열렸을 때만 보이는 피드 앱”이 아니라, 백엔드 이벤트와 모바일 푸시가 결합된 실시간 경보 앱으로 구현해야 한다. 

정책과 안전장치
외부 소스 수집은 반드시 공식·허용된 API 우선으로 가야 한다. X는 개발자 가이드라인에서 비공식 수단과 스크래핑을 금지하고 공식 API 사용과 rate limit 준수를 요구한다. Reddit도 API access rules를 존중하라고 명시한다. 따라서 KeyP의 source adapter는 기본적으로 YouTube/X/Reddit의 공식 경로를 우선 사용하고, Instagram/TikTok는 연결형·승인형 범위에서만 쓰며, “데이팅앱 역공학” 같은 방향은 제품 핵심에서 빼는 편이 가장 안전하다. 외부 연동 범위를 넓히고 싶다면 향후 공식 파트너 API가 있는 경우에만 adapter를 추가하는 구조로 열어두면 된다. 

특히 KeyP 내부에 사람-사람 상호매칭이 들어가는 순간, 모더레이션은 기능이 아니라 필수 인프라가 된다. Apple은 UGC·소셜 앱에 objectionable content filtering, offensive content reporting, abusive user blocking, 운영자 연락수단 공개를 요구하고, Google Play는 약관 동의, UGC 정책 명시, 신고·차단, 지속적 모더레이션을 요구한다. 그러므로 KeyP에는 최소한 신고, 차단, 뮤트, 연령 게이트, 운영자 연락처, 정책 동의, 위반 콘텐츠 큐, 모더레이터 대시보드가 반드시 들어가야 한다. 

사람 찾기 기능은 “정확한 상대를 외부 웹에서 캐내기”가 아니라 “관심사 기반 상호매칭”으로 번역해야 한다. 예를 들어 여행·교류·데이트 계열 요청이라면, KeyP는 외부 SNS에서 개인을 발굴해내는 대신 사용자 본인이 직접 등록한 의도, 관심, 도시 단위 위치, 시간 범위, 만남 목적, 공개 범위를 기반으로 내부 후보를 랭킹하고, 양쪽이 같은 모드에 opt-in 했을 때만 연결 링크를 보여주게 해야 한다. 그전에는 정확한 연락처·정확한 위치를 숨기고, 서로가 수락했을 때만 다음 단계로 넘어가는 state machine이 필요하다. 이 접근은 reciprocal recommendation의 원리에도 맞고, 앱스토어 정책 관점에서도 훨씬 방어적이다. 

초강력 프롬프트
아래 프롬프트는 Replit의 최신 모바일 앱 제작 흐름, Expo 기반 네이티브 앱 구조, 공식 API 우선 수집, 비용 민감형 소스 라우팅, 피드백 학습, 내부 opt-in 상호매칭을 한 번에 반영하도록 작성했다. 특히 YouTube는 push-first, X는 cost-aware hot source, Reddit은 community/live source, Instagram/TikTok는 creator-opt-in/연결형 source로 다루도록 유도했다.
